## Database Schema

```sql
repos(repo_id PK, name, created_at)
entities(entity_id PK, token_hash SHA-256, created_at)
artifacts(artifact_id PK SHA-256, repo_id FK, content BLOB, artifact_type, author, created_at, parent_id)
artifact_roles(artifact_id, role_name) -- envelope: which roles can read
role_assignments(entity_id, role_name, repo_id, granted_at, expires_at)
-- Blobs (separate system):
blobs(blob_id PK SHA-256, content BLOB, author, created_at)
blob_tags(blob_id, tag)
blob_permissions(blob_id, role_name)
```

`_system` repo auto-created for privilege management.

## Transport Topology

```
  dens (kernel pipe) ──→ proxy (TCP REQ) ──→ store_service (REP)  # all CRUD, auth, roles
  dens (kernel pipe) ──→ proxy (TCP REQ) ──→ vocation (REP)      # den-to-den, peer requests
  change.c (PUB)     ──→ dens (SUB via proxy)                    # artifact change events
  den (PUB via proxy) ──→ subscribers (SUB)                       # den-specific notifications
  den (REP via proxy) ──→ clients (REQ)                           # den API (serve_recv/serve_send)
  village (REP)       ──→ remote villages                         # clone, relay
```

All I/O from sandboxed dens goes through a kernel pipe proxy. The proxy process (unsandboxed) handles actual TCP connections. Dens never touch the network directly.

All dens get 4 bedrock sockets (via proxy): SUB (listen), REQ (store), PUB (notify), REP (serve).

### Transport Encryption

REQ/REP channels support message-level AEAD (XChaCha20-Poly1305) using a transport key derived from the bedrock key. `strata_send`/`strata_recv` auto-detect: if the message has AEAD magic bytes, decrypt; otherwise treat as plaintext. When no bedrock key is set (`STRATA_BEDROCK_KEY`), transport is plaintext.

PUB/SUB stays plaintext (topics must be readable for subscription filtering; payloads are metadata only). Relay proxies forward raw bytes (transparent).

Each den gets a local SQLite database (loaded from store on start, saved back on stop).

### Kernel Pipe Proxy

Every `strata_den_spawn()` creates a double-fork:
1. **Proxy process** (child): runs `strata_msg_proxy_run()`, handles all TCP on behalf of the den
2. **Den process** (grandchild): calls `strata_msg_set_kernel(pipe_fd)`, all `strata_req_connect`/`strata_send`/`strata_recv` go through the pipe

The proxy enforces capability permits: only endpoints declared via `strata_den_add_peer()`, `strata_den_js_register()` etc. are allowed. Unknown endpoints are rejected.

## Writing Dens

A den is a reactive Prolog state. Program clauses (from `load()`) are code — always fresh from source. Runtime facts (from `updateFact`, `engine.addClause`, Prolog `assert/1`) are state — persisted automatically.

### Level 1: Pure Prolog handlers

```javascript
var den = createDen({name: "greeter"});
den.load(
  "handle(say) :- message(M), send(respond, echo(M)).\n" +
  "handle(status) :- send(respond, ok).\n"
);
den.run();
```

The runtime asserts `from(F)`, `message(M)`, `request(action)` as ephemeral facts before querying `handle(Action)`. `send(respond, Term)` becomes the JSON response. `send(publish, topic(payload))` becomes a PUB notification.

### Level 2: JS handlers + Prolog state

```javascript
var den = createDen({name: "counter"});
den.load("count(0).\n");

den.on("increment", function(req) {
  var q = den.engine.queryFirst(
    PrologEngine.compound("count", [PrologEngine.variable("N")])
  );
  var n = q.args[0].value;
  den.updateFact("count", [], [n + 1]);
  den.bump();
  return {ok: true, count: n + 1};
});

den.run();
```

### Level 3: Privileged vocation den

```javascript
// Requires: strata_den_set_privileged(host, "anthropic", 1)
// Gets bedrock.fetch() — direct curl, no proxy hop
var den = createDen({name: "anthropic"});

den.on("ask", function(req) {
  var raw = bedrock.fetch(JSON.stringify({
    url: "https://api.anthropic.com/v1/messages",
    method: "POST",
    headers: ["content-type: application/json", "x-api-key: " + api_key],
    body: JSON.stringify({model: "claude-sonnet-4-6", messages: req.messages})
  }));
  return JSON.parse(raw);
});

den.run();
```

### createDen() API

```javascript
var den = createDen({
  name: "myden",           // den name (for logging)
  subscribe: ["topic/"],   // SUB topic filters
  maxRequests: 0           // 0 = unlimited, >0 = exit after N requests
});

den.load(text)             // Load Prolog text (program clauses, not persisted)
den.on(action, fn)         // Register JS handler: fn(req) → response object
den.run()                  // Enter serve loop (invisible)
den.save()                 // Explicit save (also called on exit)
den.restored               // true if state was loaded from previous run

// Pass-through from den engine:
den.engine                 // PrologEngine instance
den.updateFact(f, keys, vals)  // Retract-before-assert
den.bump()                 // Trigger reactive recomputation
den.createQuery(goalFn)    // Reactive query (auto-recomputes)
den.createQueryFirst(goalFn)
den.onUpdate(fn)           // Reactive effect
den.poll()                 // Drain SUB → facts → bump
```

### State Persistence

- `_prolog_state` table in per-den SQLite stores runtime facts as JSON
- Program clauses (from `den.load()`) are NOT persisted — they reload from source
- Runtime facts (from `updateFact`, `addClause`, Prolog `assert/1`) ARE persisted
- Saved automatically when `den.run()` exits
- Restored automatically when `createDen()` is called (check `den.restored`)

### JS Globals (available in all JS dens)

- `PrologEngine` — Full Prolog interpreter: `atom()`, `variable()`, `compound()`, `num()`, `new PrologEngine()`, `.addClause()`, `.retractFirst()`, `.query()`, `.queryFirst()`, `.queryWithSends()`
- `parseTerm(text)` / `parseClause(text)` / `parseProgram(text)` — Parse Prolog text to term objects
- `loadString(engine, text)` — Parse + load clauses into an engine. Returns clause count.
- `termToString(term)` / `listToArray(term)` — Term serialization
- `createSignal(initial)` / `createMemo(fn)` / `createEffect(fn)` — Reactive primitives
- `createReactiveEngine(engine)` — Reactive Prolog: `.createQuery()`, `.createQueryFirst()`, `.bump()`, `.act()`, `.onUpdate()`
- `createDenEngine(engine)` — SUB→facts bridge: `.poll()`, `.processEvent()`, `.updateFact()`
- `createDen(opts)` — Full den runtime: load, handlers, serve loop, persistence

### Bedrock API

Available in both native C and JS dens:

- `bedrock.log(msg)` — stderr
- `bedrock.request(json)` — REQ to store, returns response
- `bedrock.request(json, endpoint)` — REQ to peer den/vocation (endpoint must be declared via `strata_den_add_peer`)
- `bedrock.subscribe(filter)` / `bedrock.receive()` — SUB events
- `bedrock.publish(topic, payload)` — PUB notification
- `bedrock.serve_recv()` / `bedrock.serve_send(resp)` — REP API
- `bedrock.db_exec(sql)` — execute SQL on per-den local SQLite (returns rows changed)
- `bedrock.db_query(sql)` — query local SQLite, returns JSON array of row objects
- `bedrock.fetch(json)` — **privileged dens only**: direct HTTP via curl. Input: `{url, method, headers, body}`. Returns: `{ok, status, body}`

## Vocations

Vocations are privileged dens that provide tools to other dens. Same bedrock, same lifecycle, same den host. The only difference: `strata_den_set_privileged(host, name, 1)` skips the OS sandbox.

- **anthropic** — Claude API proxy (JS den, privileged): `ask`, `models`, `discover`
  - Owns the API key (from spawn event, persisted in Prolog facts)
  - Uses `bedrock.fetch()` for direct curl — no messenger hop
  - Key source: `private/config.json` (git-crypt encrypted) → village launcher reads → passes via spawn event
  - Key never leaves the den process
- **code-smith** — file I/O + shell (C vocation): `read`, `write`, `exec`, `glob`, `grep`, `ls`, `discover`
  - Path-sandboxed to `--root`, optional `--readonly` mode
  - Plain text via `talk` becomes shell commands; JSON messages dispatch directly
- **cobbler** — C source validator via TCC (C vocation): `compile`, `compile_file`, `discover`
  - Uses vendored TCC for in-memory compilation/validation
  - `compile`: inline C source → validates, reports size and entry points (serve/on_event)
  - `compile_file`: path to `.c` file → validates, reports size and entry points
- **messenger** — HTTP client via libcurl (C vocation): `fetch`, `discover`
  - General-purpose HTTP. For Anthropic API, prefer the anthropic JS den instead (avoids curl/SecureTransport fork issues on macOS)
  - Limits: 1MB request body, 4MB response body, 120s timeout

## Security Model

### Sandbox (Default)

All dens are sandboxed by default. Both native C (TCC) and JS (QuickJS) dens get OS-level sandboxing:

- **macOS**: Seatbelt — denies `process-fork`, `process-exec`, `network-outbound`, `network-bind`
- **Linux**: seccomp-bpf — allowlist of syscalls (read, write, poll, mmap, etc.), all others killed

Sandbox is applied AFTER `bedrock_setup()` establishes sockets via the kernel pipe proxy. Dens can only communicate through pre-established bedrock sockets. They cannot:
- Open new network connections
- Fork or exec processes
- Access the filesystem beyond their local SQLite

### Privileged (Explicit)

Vocations that need network, fork, or exec capabilities are marked privileged:

```c
strata_den_set_privileged(host, "anthropic", 1);
```

Privileged dens skip the OS sandbox. They still go through the kernel pipe proxy for bedrock sockets, but they can also use `bedrock.fetch()` for direct HTTP. Use sparingly — every privileged den is a potential attack surface.

### Credential Management

API keys live in `private/config.json` (git-crypt encrypted at rest):

```json
{
  "strata": { "key": "..." },
  "anthropic": { "key": "sk-ant-..." }
}
```

The village launcher reads the key and passes it to the privileged den via the spawn event. The key is stored in the den's Prolog fact base (persisted in per-den SQLite). It never leaves the privileged den process.

Precedence: `ANTHROPIC_API_KEY` env > `--api-key` flag > `private/config.json`

## Anti-Patterns

- **Don't write SQL tables for den state.** Use Prolog facts + `createDen()` persistence. Runtime facts are saved/restored automatically.
- **Don't write serve loops.** Use `den.run()`. The loop, JSON parsing, and dispatch are handled by the runtime.
- **Don't store API keys in agent den memory.** Use a privileged vocation den that owns the credential.
- **Don't mark dens privileged to work around proxy issues.** Fix the proxy. Every privileged den is a security hole.
- **Don't use hex byte array embeds.** Run `scripts/gen_js_embed.py` to generate readable C string literals.
- **Don't call curl from forked C vocations on macOS.** SecureTransport uses ObjC runtime which is not fork-safe. Use `bedrock.fetch()` in a privileged JS den instead.
- **Don't use standalone C vocation binaries for new vocations.** Write them as JS dens with `privileged=1`. Same bedrock, same lifecycle, one fewer layer.

## Embed Pipeline

JS sources from `../wyatt/src/` and `dens/lib/` are embedded as C string literals in header files. The pipeline:

1. Edit JS source (e.g., `dens/lib/den-runtime.js` or `../wyatt/src/prolog-engine.js`)
2. Run `python3 scripts/gen_js_embed.py`
3. Generates `vendor/prolog/prolog_js_embed.h` (reactive, prolog-engine, parser, loadString, reactive-prolog) and `vendor/prolog/den_engine_js_embed.h` (den-engine, den-runtime)
4. Rebuild: `cmake --build build`

The embeds are readable C strings — auditable and diffable, not hex blobs.

## Key Conventions

- **C11**, no C++. All code is plain C.
- **Lightweight TCP transport:** length-prefixed frames over TCP. No external messaging library. Kernel pipe proxy for sandboxed dens.
- **Role-filtered access:** artifact list/get queries JOIN on artifact_roles + role_assignments. Entity sees only what their roles permit.
- **Fork isolation:** every den runs in its own process via double-fork (proxy + den). No threads for den execution.
- **JSON protocol:** store_service speaks JSON over REP. Actions: put, get, list, init, repo_create, role_assign, role_revoke, privilege_grant, privilege_revoke, privilege_check, entity_register, entity_authenticate, blob_put, blob_get, blob_find, blob_tag, blob_untag, blob_tags.
- **Vocation pattern:** vocations are privileged dens, not a separate subsystem. Agent carries role, vocation provides hands. Same REP socket, same JSON actions, same `talk` command.
- **Den = reactive Prolog state:** program clauses are code (from source), runtime facts are state (persisted). Messages are invisible infrastructure — facts flow in (accept), signals flow out (send/2).
- **Town-hall as message board:** inter-agent communication via shared artifact repo. Agents post to and read from the board.
- **Per-den local SQLite:** each den gets its own db, loaded from village store (base64 blob) on start, saved back on stop. Runtime Prolog facts also stored here.
- **Test pattern:** fork store_service in child process, wait for readiness probe, run assertions, cleanup via atexit + signal handlers.
- **Tests use relative paths** to `dens/` — CMakeLists.txt sets WORKING_DIRECTORY to source root.

## Hard Rules

- No agent runs outside a sandbox (fork + OS sandbox) unless explicitly privileged.
- No plaintext at rest (AEAD-encrypted blobs).
- Transport encryption available via `STRATA_BEDROCK_KEY` (auto-detected, backward compatible with plaintext).
- Capability injection, not request. Host decides what den gets.
- Immutable audit trail. No rebase, no history rewrite.
- Storage-agnostic core. No direct SQLite calls in business logic (store interface only).
